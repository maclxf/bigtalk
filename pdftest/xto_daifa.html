包裹代发「Daifa API 开单」技术文档 (v1.0)

概述

本文档详细说明了通过易多客（edockey.com）实现包裹代发API在线开单的技术规范、字段要求及相关注意事项

请求规范

基本规范

  • 请求方式：POST
  • 请求URL：http://api.edockey.com/v1/xto_daifa

请求参数

请求参数包括固定参数和业务参数。固定参数是每次API请求都必须传入的参数。将请求参数统一做json_encode处理，对请求的HEADER设为'Content-type: application/json;charset=UTF-8'。

固定参数

  1. access_key_id： 这是易多客分配给调用者的访问ID，是公开的

  2. timestamp： 发起API请求时的UNIX时间戳，这是为了防止重放攻击

  3. signature： 发起API请求时的签名，易多客会通过该签名来验证所发起的请求是否有授权。具体签名算法见下面的说明

业务参数

包裹代发API开单，业务参数为一个二维数组，各字段及说明如下：

  1. access_key_id (string)这是易多客分配给调用者的访问ID，是公开的
  2. timestamp (string)发起API请求时的UNIX时间戳
  3. signature (string)发起API请求时的签名
  4. channel_code (string)在已开通的渠道列表接口中返回的渠道代码,详情请看特别说明
  5. total_weight (float)包裹总重量
  6. total_price (float)包裹总价值
  7. order_number (string)包裹商家订单号
  8. tracking_number 开单后返回的末端物流号，请传’‘
  9. shipping_amount (float)运费
  10. purchaser_name (string)支付者姓名
  11. purchaser_cardid (string)支付者身份证号
  12. consignee_name (string)收件人姓名
  13. consignee_phone (string)收件人电话
  14. consignee_email (string,可选,没有时填’‘)收件人电子邮件
  15. consignee_province (string)收件人省份
  16. consignee_city (string)收件人城市
  17. consignee_county (string)收件人地区
  18. consignee_address (string)收件人地址
  19. consignee_zipcode (string, 可选，没有时可填000000)收件人邮编
  20. sendername (string, 可选，没有时填’‘)发件人姓名
  21. senderphone (string, 可选，没有时填’‘)发件人电话
  22. senderaddress (string, 可选，没有时填’‘)发件人地址
  23. awbno (string, 可选，没有时填’‘)主单号
  24. exchange_rate (string, 可选，没有时填’‘)汇率
  25. title (string)产品名称
  26. ean (string)EAN 详情请看特别说明
  27. price (float)产品单价
  28. quantity (int)产品数量
  29. link (string, 可选，没有时填’‘)产品链接

特别说明

  1. 所有产品的ean只支持易多客系统提供的ean列表api里包含的ean,如需增加产品类型，请务必事先联系我们 ean列表api的说明文档访问地址是 http://api.edockey.com/doc/xto_ean_list.html

  2. channel_code只支持易多客系统提供的线路编码列表api里包含的渠道编码，如需开通新的渠道，如奢侈品、服装等， 请联系我们 渠道编码列表api的说明文档访问地址是 http://api.edockey.com/doc/channel_code_list.html

签名算法

易多客提供的API，使用对称加密算法来实现签名，具体流程如下：

  1. 将所有请求参数构造成一个数组，并将数组按照a-z的顺序进行自然排序（注意：access_key_id 和 timestamp 两个参数也要参与签名）
  2. 把请求参数用 & 进行连接（注意：需要按照RFC3986标准对字符串进行编码，如：空格会被编码成 %20）
  3. 在上述生成的字符串后面使用 & 连接易多客分配给你的 access key secret，这是最终待签名的字符串
  4. 使用 MD5算法 对上述待签名字符串进行加密，生成的加密字符串即为本次请求的签名

$signature = $this->signature($params);                  //这个$signature就是你的签名

public function signature(array $params) {               //生成签名的方法
    if (!$params) {
        return '';
    }

    if (!is_array($params)) {
        return '';
    }

    // 提取 access key id
    $access_key_id = $params['access_key_id'];
    if (!$access_key_id) {
        return '';
    }

    // 获取 access key secret
    $access_key_secret = $this->access_key_secret;
    if ($access_key_secret === '') {
        return '';
    }

    // 按照键名由a-z进行排序
    $params = $this->_sort($params);

    // 生成待签名的字符串
    // 需要按照RFC3986标准对字符串进行编码，如：空格会被编码成%20
    $strToSign = $this->_build_params($params);

    // 在待签名后面添加 access key secret，形成最终待签名的字符串
    $strToSign = "{$strToSign}&access_key_secret={$access_key_secret}";

    // 使用 md5 方式进行签名，获得最终的签名字符串
    $signature = md5($strToSign);

    // 返回签名后的字符串
    return $signature;
}

/**
 * 对数组按键名由a-z进行排序
 * @param  array  $params 要排序的数组
 * @return array          排序之后的数组
 */
private function _sort(array $params) {
    ksort($params);
    reset($params);
    return $params;
}

/**
 * 按照 RFC3986 标准来构建待签名字符串
 * RFC3986标准将使得空格编码成%20，而不是+
 * @param  array  $params 用来构造待签名字符串的参数
 * @return string
 */
private function _build_params(array $params) {
    return http_build_query($params, '' , '&', PHP_QUERY_RFC3986);
}

**特别注意**
php 5.4以下http_build_query方法中请不要传第四个参数，即PHP_QUERY_RFC3986。此时应这样写

private function _build_params(array $params) {
    $signature_str = http_build_query($params, '' , '&');
    $signature_str = str_replace("+", "%20", $signature_str);
    return $signature_str;
}

返回结果

返回结果类型为json，包含4个值其中status，code，message，data。 status： 返回的状态说明success 或 failed code： 返回的状态码,其中200表示执行成功 message： 状态信息说明 data： 响应返回的数据数组（api的开单结果）

返回示例

{
    "status": "success",
    "code": 200,
    "message": "开单成功",
    "data": {
        "tracking_number": "XXXXXX",          //清关公司返回的末端物流号
        "pdf_url": "XXXXXX"                   //面单下载地址
    }
}

常见报错解释及其可能性

大多数情况下，当接口报错的时候，数据传递错误的可能性是最大的，建议先阅读报错信息，再针对性的去处理。 以下是常见的报错实例。

  1. 18位 解释：身份证位数不正确

  2. 订单号xxx已存在 解释：xxx这个订单已经成功开过单了

  3. 检验位不正确 解释：身份证号码和人名不对应

  4. 此ean未备案 解释：包裹中包含没有经过清关公司备案的商品

  5. Column ‘purchaser_name’ cannot be null 解释：遇到这种情况，从字面上理解应该是支付者姓名不能为空，但是往往是正常传递了支付人姓名的，所以请仔细查看整个数据，是否有特殊字符、空等情况

  6. Out of order limit. 解释：产品价值超过2000人民币，请注意，这里的价值是指运费加产品的备案价值

  7. ID No is invalid 解释：身份证无效

  8. 三字代码获取失败 解释：收件地址有问题。之前遇到过地区和城市传反的情况

  9. 圆通单号获取失败 解释：这种情况下，可以尝试其他包裹的开单，如果依旧报这个错误，可能就是清关公司的圆通单号池耗尽，这时候请联系我们，我们将会告知清关公司尽快补充

  10. 请求参数不能为空 解释：调用接口时，请求的json字符串解析出错，请检查json字符串重是否包含非法字符（tab，换行符等）
